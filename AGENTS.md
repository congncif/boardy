# Boardy Project

**Type**: Swift iOS Framework  
**Architecture**: Microservice-inspired modular orchestration  
**Language**: Swift 5.9+  
**Platform**: iOS

## Project Overview

Boardy is a lightweight orchestration framework that adapts microservices principles for iOS app architecture. It enables modular, flow-driven applications through two core concepts:

- **Board**: A self-contained, stateless business unit (microservice-like). Activated via `activate()`.
- **Motherboard**: Central orchestrator managing board lifecycle and workflow coordination.

Boards communicate through a unified protocol layer and can be dynamically composed, enabling plug-and-play modularity.

## Architecture Principles

### 1. Stateless Boards (Preferred)
- Boards should NOT retain internal state
- Lifecycle managed automatically by Motherboard
- Call `complete()` only if using stateful pattern (not recommended)

### 2. Type-Safe Communication
- Input/Output types enforce contracts between boards
- Use `IOInterface` for type-safe board interactions
- Generated by Xcode templates

### 3. Separation of Concerns
- **Board**: Flow orchestration glue (stateless)
- **Controller**: Business logic (any architecture: MVC, VIP, Clean)
- Communication via EventBus and delegates

## Key Patterns

### Board Structure
```swift
final class SomeBoard: ModernContinuableBoard, GuaranteedBoard, 
                       GuaranteedOutputSendingBoard, GuaranteedActionSendingBoard {
    typealias InputType = SomeInput
    typealias OutputType = SomeOutput
    typealias FlowActionType = SomeAction
    typealias CommandType = SomeCommand
    
    func activate(withGuaranteedInput input: InputType) {
        // Build controller, show UI, wire up buses
    }
}
```

### Motherboard Flow Registration
```swift
motherboard.registerFlowSteps(.pubBoardA ->> .pubBoardB ->> .pubBoardC)
```

### ContinuousBoard (Sub-orchestrator)
Wraps internal Motherboard for complex multi-step flows. Appears as single board to parent.

## Code Organization

- **Builder Pattern**: Used for dependency injection in Full UI Boards
- **Plugins**: Modular features packaged as plugins with PluginLauncher
- **Templates**: Xcode templates available for scaffolding (MVC, VIP, SwiftUI Full UI)

## Development Guidelines

### Testing Strategy (per RTK workflow)
- **Full TDD**: Core orchestration logic, board activation flows, plugin initialization
- **Test-after**: Individual board implementations, controller logic, UI wiring
- **No tests**: Type definitions, configuration

### Build Verification
- Use `swift build` or Xcode's build for type checking
- Run targeted tests: `swift test --filter TestName`
- Full suite before completion: `swift test`

### Naming Conventions
- Boards: `{Feature}Board` (e.g., `PaymentBoard`, `ShoppingCartBoard`)
- Inputs: `{Feature}Input`
- Outputs: `{Feature}Output`
- Actions: `{Feature}Action`
- Commands: `{Feature}Command`

## Key Files to Know

- `Boardy/Core/`: Framework core (Board, Motherboard, protocols)
- `Boardy/Plugins/`: Plugin system implementation
- Xcode templates in project for scaffolding

## Common Tasks

### Creating New Board
1. Use Xcode template (recommended)
2. Define Input/Output types
3. Implement `activate()` method
4. Wire up controller delegate
5. Register with Motherboard or PluginLauncher

### Adding New Feature Module
1. Create as Plugin (LauncherPlugin conformance)
2. Register boards in plugin's board producer
3. Install plugin via PluginLauncher

### Chaining Flows
Use `flow.addTarget()` to listen to board outputs and chain next activation.

## Notes

- Boards installed into root context (UIViewController/UIWindow)
- Gateway Barriers can perform pre-checks before activation
- Dynamic registration via BoardProducer preferred over static
